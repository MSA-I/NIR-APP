## מסקנה

סדר העבודה המחייב הוא P0→P4. אין להתחיל בשינויי UI לפני שחוזה התפקידים שאושר נאכף במסד הנתונים ובנתיבים.

הבריף שאושר בשיחה הוא חוזה היעד:

- `owner` — ניהול, אישור ובקרה, כולל מסלול חירום מבוקר.
- `office` — מנהל רכש.
- `supplier` — ספק מבודד למידע שלו בלבד.
- `accountant` — הנהלת חשבונות תפעולית.
- `payer` ו־`kitchen` נשמרים כפי שהם עד החלטה נפרדת; אין להרחיב או לצמצם אותם בשקט.

החוזה הזה שונה מההגדרות הקיימות ב־`PRODUCT.md` וב־`ARCHITECTURE.md`, שבהן `office` הוא power user פיננסי ו־`accountant` הוא רו״ח חיצוני לקריאה. לכן P0 מתחיל בעדכון מקור האמת ורק אחר כך בשינוי קוד. אין לשנות את enum `user_role`.

## קדם־תנאים לביצוע

- לפתוח worktree יישום מבודד מה־SHA שיאושר בתחילת הביצוע.
- worktree הביקורת הנוכחי כולל שישה שינויי harness/fixture ואת הדוחות; אין להשתמש ב־`git add .` או להפוך אותו בשקט לבסיס פיתוח.
- להעביר את תיקוני ה־harness שאושרו באמצעות allowlist והנד־אוף נפרד.
- לא לערוך מיגרציות שכבר הוחלו. כל שינוי DB ייכנס במיגרציה חדשה אחרי `0024`, לאחר בדיקת ancestry ומספור.
- לשמר את כללי המודל הכספי: N:M allocations, יתרות מחושבות, מחיקה רכה ו־audit עם סיבה.

## תלות בין השלבים

```text
P0 — חוזה capabilities והרשאות
  ├─ P1A — כספים, הנהלת חשבונות ומסלול חירום
  ├─ P1B — הגשת מחירון חודשית
  └─ P1C — מסע רכש mutable
          ↓
P2 — יעילות, מובייל ונגישות

          ↓

```

שלושת זרמי P1 יכולים להתבצע במקביל לאחר סגירת P0, אך חייבים להתמזג לפני P2.

# P0 — לקבע ולאכוף את חוזה התפקידים

## תוצאה נדרשת

מסמכי המוצר, תוויות התצוגה, דף הבית, הניווט, החיפוש, route guards, ‏RLS ו־RPC מתארים ואוכפים אותה מטריצת יכולות.

## עבודה

1. לעדכן תחילה את `PRODUCT.md`, ‏`docs/ARCHITECTURE.md` ו־`docs/OPEN-DECISIONS.md` במטריצה מפורשת של צפייה, יצירה, אישור וביצוע לכל תפקיד.
2. במיגרציה חדשה:

   - לאפשר ל־`accountant` ביצוע או רישום של תשלום שאושר, התאמות בנק ומעברי זיכוי.
   - להגביל אותו לחשבוניות מאושרות ולהקשר הרכש המינימלי הדרוש לביצוע החשבונאי.
   - להסיר מ־`office` ביצוע תשלום, התאמות בנק, דוח רו״ח ו־audit פיננסי שאינם חלק מניהול רכש.
   - להשאיר את פקודת התשלום הרגילה חסומה ל־`owner` ולספק מסלול חירום נפרד בלבד.
   - אם מסלול החירום ממומש, לדרוש אימות טרי, סיבה, idempotency ו־audit מסוג נפרד.
   - לעדכן policies של invoices, allocations, views וקשרי חשבונית. הסתרת UI אינה גבול הרשאה.

3. ליישר את הלקוח לאותו חוזה: תוויות, `homeFor`, ‏NAV, route guards ותוצאות החיפוש.

## קובצי ושכבות יעד

- מסמכים: `PRODUCT.md`, ‏`docs/ARCHITECTURE.md`, ‏`docs/OPEN-DECISIONS.md`.
- DB: מיגרציה חדשה אחרי `0024`; ‏`0023_p1_financial_command_boundaries.sql` משמשת מקור להשוואה בלבד.
- מעטפת לקוח: `src/App.tsx`, ‏`src/components/Layout.tsx`, ‏`src/auth/AuthContext.tsx`, ‏`src/lib/status.ts`, ‏`src/components/GlobalSearch.tsx`.
- מסכים: `PayerQueue.tsx`, ‏`Payments.tsx`, ‏`Bank.tsx`, ‏`Credits.tsx`, ‏`Invoices.tsx`, ‏`InvoiceDetail.tsx`, ‏`Reports.tsx`.
- בדיקות: `supabase/tests/p1_financial_commands.sql`, ‏`p1_concurrency.sql`, ‏`scripts/check-p0-security.ps1`, ‏`scripts/check-browser-smoke.cjs`.

## שער קבלה

- `office` משלים רכש אך נחסם ב־UI, route, RPC ו־RLS מפעולות כספיות שאינן שלו.
- `accountant` רואה רק חשבוניות מאושרות, מבצע תשלום מאושר ומטפל בבנק ובזיכוי, אך אינו משנה הזמנה או מחירון.
- `payer` ו־`kitchen` עוברים מטריצת רגרסיה ללא שינוי חוזה.
- `supplier` רואה רק את הספק שלו; בדיקות ספק מתחרה ודייר שני עוברות.
- `owner` חסום מפקודת התשלום הרגילה; מסלול החירום, אם קיים, דורש אימות טרי וסיבה ומייצר audit.
- direct writes, חציית דייר ומחיקה קשה נשארים חסומים.
- `npm.cmd run build` ובדיקות DB ממוקדות עוברים לפני P1.

# P1 — לסגור את משימות הליבה

## P1A — הנהלת חשבונות ובעלים

- להפוך את `/pay` לתור עבודה גם ל־`accountant`, תוך שימוש חוזר ב־`PayerQueue.tsx` ושמירה על בקרות האישור.
- לחשוף ל־`accountant` ייבוא, התאמה והסרת התאמה בבנק ואת מעברי הזיכוי.
- להפוך מדדי דוח לקישורים מסוננים לתשלום, בנק, זיכויים וחריגים.

- להוסיף לבעלים UI נפרד למסלול החירום; לא להלביש אותו על הפעולה הרגילה.

יעדים עיקריים: `PayerQueue.tsx`, ‏`Payments.tsx`, ‏`Bank.tsx`, ‏`Credits.tsx`, ‏`Reports.tsx`, ‏`PaymentRequests.tsx` והמיגרציה של P0.

## P1B — הגשת מחירון חודשית

- להוסיף ledger של הגשות עם `org_id`, ‏`supplier_id`, חודש יעד, revision, שם ונתיב קובץ, checksum, סטטוס, זמני הגשה ומוני accepted/rejected/unchanged.
- תיקון לאותו חודש יוצר revision חדש ואינו דורס קבלה קודמת.
- לשמור קליטה אטומית: קובץ זמני → parsing מלא → RPC יחיד → מחירים, `price_history` והגשה.
- להציג לספק קבלה חודשית, סטטוס והיסטוריית הגשות.
- להשתמש ב־`src/lib/importSheet.ts` הקיים במקום parser ‏XLSX כפול ב־`SupplierPrices.tsx`.
- להציג בתוך הדיאלוג שורות ומוצרים לא מוכרים ושגיאת קובץ פגום; לנקות מיד מצב “מעדכן”.
- לא ליצור מוצר קטלוג חדש על סמך שם שספק העלה ללא תהליך מיפוי או אישור של מנהל הרכש.

יעדים: מיגרציה חדשה נפרדת, Storage תחת `{org_id}/`, ‏`src/pages/SupplierPrices.tsx`, ‏`src/lib/importSheet.ts`, ‏`src/lib/types.ts`, ובמידת הצורך `PriceLists.tsx` לתצוגת מנהל הרכש.

## P1C — מסע רכש mutable

להריץ תחילה מסע מבודד אמיתי:

הזמנה → Auto Save → השוואת ספקים → finalize → שליחה → קבלה מלאה/חלקית/פגומה → חשבונית → אישור → חריג/זיכוי.

אין לשכתב מסכים לפני שנמצא חיכוך חוזר. יעדים אפשריים רק לפי תוצאת המסע: `NewOrder.tsx`, ‏`Orders.tsx`, ‏`Receiving.tsx`, ‏`InvoiceNew.tsx`, ‏`InvoiceDetail.tsx`, ‏`Credits.tsx` וה־RPC הקיימים.

## שער קבלה

- תשלום תומך partial ו־N:M; retry מחזיר אותה תוצאה ושתי פעולות מקבילות אינן מכפילות תשלום.
- התאמה והסרת התאמה בבנק מחייבות סיבה ומייצרות audit.
- מעבר זיכוי משנה יתרה פעם אחת בלבד.
- הגשת מחירון מציגה חודש, אסמכתה, סטטוס והיסטוריה; כשל אינו משאיר מחירים חלקיים.
- קובץ פגום ומוצר לא מוכר מקבלים משוב מעשי.
- מסע הרכש המלא עובר עם DB לפני/אחרי ו־audit.
- כל בדיקות השלילה של P0 נשארות ירוקות.

# P4 — קבלה משולבת ורולאאוט

## המסע המשולב

ספק מגיש מחירון → מנהל רכש מזמין → מתקבלת סחורה → חשבונית נבדקת ומאושרת → בעלים מאשר דרישת תשלום → הנהלת חשבונות מבצעת → בנק מותאם → זיכוי מקוזז → הנהלה רואה את התוצאה בדשבורד ובדוח.

## שער מקומי

- worktree נקי, provenance מתועד ומספור migrations מפויס.
- reset והתקנה מ־`0019` ועד המיגרציות החדשות עוברים.
- `npm.cmd run quality` מסתיים ב־exit code ‏0 על fixture טרי.
- כל מטריצות role/RLS/RPC, browser, viewports ונגישות עוברות.
- DB לפני/אחרי, audit, screenshots ו־exports נשמרים ללא secrets.
- אין P0/P1 פתוחים; P2/P3 שנותרו דורשים waiver מפורש.

## רולאאוט

- local PASS אינו אישור deploy.
- staging דורש אישור נפרד, migration dry run ומסעות smoke לכל תפקיד.
- production דורש אישור נוסף, snapshot או תוכנית forward-fix, ניטור והוכחה חזותית בפועל.
- לאחר כל שלב יש לעדכן את `docs/PROGRESS.md`, ‏`ARCHITECTURE.md`, ‏`OPEN-DECISIONS.md` ודוחות הקבלה.

## סדר מסירה מומלץ

1. מסמך capability matrix מאושר ומיגרציית P0 עם בדיקות שלילה.
2. התאמת routes/NAV/search לתוצאות P0.
3. שלושת זרמי P1 במקביל, כל אחד ב־worktree ובעלות קבצים מפורשת.
4. אינטגרציה, P2 משותף וביקורת עיצוב/נגישות.
5. P3 מדידה והקשחה.
6. P4 מקומי מלא; רק לאחר אישור נפרד מעבר ל־staging ול־production.
