#### הסתייגות לפריט 6 — התייקרויות

`supplier_products.previous_price` (`0001_init.sql:108`) הוא עמודה בת-חריץ-אחד: היא לוכדת רק את השינוי **האחרון**. הקירוב הנכון:

```
price_effective_date >= היום-30 יום  AND  current_price > previous_price
```

`price_history` (`0001_init.sql:118-126`) שומרת היסטוריה מלאה, אך רק `price` (`:122`) ו-`effective_date` (`:123`) — "התייקר בתוך חלון" מדויק היה דורש שאילתת `lag()`. **להשתמש בקירוב של `supplier_products` בגרסה ראשונה ולתעד זאת.**

תיקון נדרש ב-`Dashboard.tsx:37`: להוסיף `.gte('price_effective_date', last30d).not('previous_price','is',null)`.

שימו לב שהמדד הוא **ספקים**, לא מוצרים: `new Set(rows.map(r => r.supplier.name)).size`.

### 3.2 קומפוננטה חדשה — אזור "דורש טיפול היום"

#### למה **לא** לעשות שימוש חוזר ב-`KpiCard`

שלוש סיבות קונקרטיות:

1. **הוא מייצר בדיוק את הפריסה האסורה.** `KpiCard` (`src/components/ui.tsx:35-47`) מרנדר `card card-pad` עם מספר `text-xl font-bold`. שמונה כאלה = שתי שורות של כרטיסים עבים בגובה ~200px. `CLAUDE.md:49` אוסר במפורש **"הפיכת כל דף לרשת כרטיסים מנופחת"** ו**"מראה תבנית-אדמין גנרית"**. בדף כבר יש **עשרה** כאלה (`:140-155`); שמונה נוספים הופכים אותו לגרוע יותר, לא טוב יותר.
2. **אוצר-המילים של הגוונים שלו הוא פיצול.** ה-`tone` של `KpiCard` הוא איחוד inline `'slate'|'green'|'amber'|'red'|'blue'` (`ui.tsx:36`) שמשמיט `violet` ומשכפל את `Tone` מ-`src/lib/status.ts:2`. אזור הטיפול צריך לדבר את אוצר-המילים המרכזי האחד.
3. **הוא אינו יכול לבטא את צורת הנתונים.** זיכויים פתוחים דורשים ספירה **וגם** ₪. חריגים דורשים "12, מתוכם 3 בחומרה גבוהה". ל-`KpiCard` יש `value: string` יחיד ו-`sub` אופציונלי.

#### המוצע: קומפוננטה חדשה אחת, כרטיס אחד, שורות צפופות

**להוסיף ל-`src/components/ui.tsx`** אחרי `KpiCard` (כלומר אחרי `:47`):

```ts
export interface AttentionItem {
  key: string;
  label: string;
  count: number | null;     // null = אין נתונים → מוצג "—", לא 0
  amount?: number | null;   // ₪, אופציונלי
  tone: Tone;               // מיובא מ-lib/status — עושה שימוש ב-.badge-* שב-index.css:34-39
  to: string;               // נתיב מלא כולל query string
  hint?: string;            // "3 בחומרה גבוהה"
}
export function AttentionZone({ items }: { items: AttentionItem[] }) { /* ... */ }
```

**מבנה:** `.card` **יחיד**, ובתוכו שורת כותרת ורשימת `<ul>` של שורות בנות שורה אחת (~40px כל אחת). שמונה שורות ≈ 340px בכרטיס אחד, לעומת ~2 שורות רשת בתוספת מרווחים עבור שמונה `KpiCard`.

מבנה שורה (RTL — תכונות לוגיות בלבד):
`[badge-{tone} count] [label] … [amount, class="num"] [ChevronLeft]`

**שתי דרגות בתוך הכרטיס האחד:**

- **דרגה א'** — פריטים עם `count > 0`, בסדר החשיבות העסקית. שורות מלאות.
- **דרגה ב'** — פריטים עם `count === 0`, מקופלים לשורה מושתקת אחת: `✓ אין חריגים · ✓ אין תשלומים באיחור`. שמונה אריחי-אפס שכולם צועקים באותה עוצמה הם ההפך מ"רגוע" (`CLAUDE.md:47`) והורסים את יעד "להבין תוך מספר שניות". כשדרגה א' ריקה — שורה מרגיעה אחת, בלי שטח דקורטיבי ריק (`CLAUDE.md:49`).

**`count === null`** (אין נתונים כלל) מוצג `—` ואינו נספר בסיכום הכותרת.

**שורת כותרת:** `דורש טיפול היום` + סיכום נגזר (`4 פריטים · ₪12,340`). בלי אנימציה, בלי glassmorphism, בלי פעולות מוסתרות מאחורי hover.

### 4.1 "מוצרים בעלי מלאי נמוך" — פריט 8. **לא לממש. לא להציג אפס.**

### 4.2 "תשלומים המיועדים לביצוע היום" ו"תשלומים באיחור" — פריטים 3 ו-4. **חלקית בלבד.**

הפריטים ניתנים למימוש **אך ורק דרך `payment_requests`, לא דרך חשבוניות.**

- **`suppliers.payment_terms`** (`0001_init.sql:72`) הוא **`text` חופשי**. הערכים בפועל הם מהצורה "שוטף + 30". **שום קוד במערכת אינו מנתח אותו.** גזירת "החשבונית באיחור" ממנו תדרוש פרסר על טקסט חופשי בעברית — כלומר להמציא כלל עסקי.
- **`payment_requests.due_date`** (`0001_init.sql:253`) הוא המקור היחיד לתאריך יעד. הוא **`nullable`**, ונקבע **ידנית** בעת יצירת הדרישה — `src/pages/PaymentRequests.tsx:136` (`due_date: dueDate || null`), מתוך שדה קלט ב-`:82`.

**המשמעות:** "תשלומים באיחור" מודד **דרישות תשלום שנוצרו וקיבלו תאריך יעד ידני**, ולא "כסף שאני חייב לספק ועבר המועד". חשבונית שמעולם לא הפכה לדרישת תשלום **אינה נראית** במדד הזה, לעולם.

**ההכרעה:** התווית חייבת לומר את האמת — **"דרישות תשלום באיחור"**, לא "תשלומים באיחור". כשאין ולו דרישה אחת עם `due_date` — **המדד מציג `—`, לא `0`**, כי `0` היה מבטיח "שום דבר לא באיחור" ואין לנו את הידע הזה. ראה הכרעה פתוחה #2.
